前面几篇主要讲应用工程：RAG、上下文、工具、Agent 和 MCP。

从这一篇开始，回到模型训练主线。

很多人知道 RLHF 这个词，也知道它和“让模型更符合人类偏好”有关。但 RLHF 不是一个单步骤算法，而是一条训练流程：

```text
预训练模型
  -> SFT
  -> 偏好数据
  -> 奖励模型
  -> PPO 强化学习
  -> 对齐后的助手模型
```

这篇文章先讲经典 RLHF 主线：SFT、奖励模型、PPO 和偏好数据分别在做什么。

## 1. 预训练模型会续写，不一定会当助手

大语言模型预训练时，核心目标通常是预测下一个 token。

它在大量文本上学习：

```text
给定前文 -> 预测下一个 token
```

这让模型学会语言、知识、代码、格式和推理模式。

但预训练模型本质上是在“续写文本”。用户问：

```text
请解释 RAG 是什么。
```

预训练模型可能续写成答案，也可能续写成对话、网页片段、论坛回复、甚至奇怪格式。

要让它更像助手，需要后训练。

RLHF 是后训练的一条经典路线。

## 2. SFT：先教模型按指令回答

SFT 是 supervised fine-tuning，监督微调。

它用人工或高质量数据构造指令和答案：

```json
{
  "instruction": "解释什么是 RAG。",
  "response": "RAG 是检索增强生成..."
}
```

训练目标仍然是预测下一个 token，但数据形式变成了“用户指令 -> 助手回答”。

SFT 让模型学会：

- 对话格式。
- 遵循指令。
- 简洁回答。
- 拒绝部分不合适请求。
- 使用特定语气。
- 输出结构化格式。

可以把 SFT 理解成：先把预训练模型拉到“助手模式”。

## 3. SFT 的局限

SFT 很重要，但它有局限。

第一，它只学习示范答案。

如果一个问题有多个候选回答，SFT 只告诉模型“这个答案是好的”，不告诉模型“另一个答案为什么更差”。

第二，示范数据很贵。

高质量人工答案需要领域专家、审核和一致标准。

第三，SFT 容易平均化。

模型会学习数据里的常见表达，但对细微偏好不敏感。

例如两个回答都正确：

```text
A：简短直接，但缺少风险提示。
B：稍长一点，包含限制条件和例子。
```

人类可能更喜欢 B。SFT 不容易表达这种偏好差异。

这就引出了偏好数据。

## 4. 偏好数据：让人比较两个回答

偏好数据通常不是“写一个标准答案”，而是“比较两个或多个回答哪个好”。

例如：

```json
{
  "prompt": "解释 RAG 为什么需要引用。",
  "chosen": "RAG 引用能让用户追溯答案来源...",
  "rejected": "RAG 引用就是给回答加几个链接..."
}
```

标注者判断 `chosen` 比 `rejected` 更好。

偏好数据能表达：

- 哪个回答更准确。
- 哪个回答更有帮助。
- 哪个回答更安全。
- 哪个回答更符合语气。
- 哪个回答引用更可靠。
- 哪个回答更少废话。

它不要求标注者从零写答案，只要求比较。

这通常比写完整示范答案更容易规模化。

## 5. 奖励模型：学习人类偏好

有了偏好数据，就可以训练奖励模型。

奖励模型输入 prompt 和回答，输出一个分数：

```text
(prompt, answer) -> reward score
```

训练目标是让 chosen 的分数高于 rejected。

直觉上：

```text
人类更喜欢的回答 -> 更高 reward
人类不喜欢的回答 -> 更低 reward
```

奖励模型不是最终助手。它更像评委，用来给候选回答打分。

训练好奖励模型后，就可以让语言模型生成回答，再根据奖励模型的分数优化语言模型。

## 6. PPO：用奖励信号优化策略

PPO 是一种强化学习算法。

在 RLHF 里，可以把语言模型看成策略：

```text
状态：prompt 和上下文
动作：生成下一个 token
轨迹：完整回答
奖励：奖励模型给回答打分
```

PPO 的目标是让模型更倾向于生成高奖励回答。

但不能让模型偏离原模型太远。否则它可能为了骗奖励模型而产生奇怪输出。

所以 RLHF 通常会加入 KL 惩罚：

```text
新模型不要离参考模型太远
```

简化目标是：

```text
提高 reward
同时限制和原模型的差距
```

这就是 PPO 在 RLHF 里的作用。

## 7. 为什么需要 KL 约束

如果只最大化奖励模型分数，会出现 reward hacking。

模型可能学会奖励模型的漏洞，而不是真正变好。

例如：

- 输出过度礼貌但没内容。
- 总是写很长，骗取“详细”偏好。
- 反复使用奖励模型喜欢的模板句。
- 在安全场景中过度拒答。

KL 约束让模型不要离 SFT 模型太远，减少训练发散。

可以理解成：

```text
奖励模型说往这个方向更好
KL 约束说不要跑太远
```

两者共同控制训练。

## 8. RLHF 的完整流程

把前面串起来：

```text
1. 预训练
   学会语言和知识

2. SFT
   学会按指令回答

3. 采样回答
   对同一 prompt 生成多个候选

4. 人类偏好标注
   比较哪个回答更好

5. 训练奖励模型
   学会给回答打偏好分

6. PPO
   用奖励模型优化助手模型

7. 评测和安全检查
   判断是否真的变好
```

这条链路的关键是：人类偏好没有直接写成规则，而是通过数据和奖励模型影响训练。

## 9. RLHF 解决了什么

RLHF 能改善：

- 指令遵循。
- 对话可用性。
- 答案风格。
- 拒绝不合适请求。
- 减少明显有害输出。
- 更符合人类对“好回答”的直觉。

它把模型从“会续写文本”推向“更像助手”。

但 RLHF 不是万能的。

它不能保证：

- 所有事实都正确。
- 不会幻觉。
- 数学推理一定可靠。
- 工具调用一定正确。
- 引用一定真实。
- 不会被越狱。

这些仍然需要 RAG、工具校验、评测、安全策略和产品边界。

## 10. 偏好数据质量决定上限

RLHF 的核心资产是偏好数据。

偏好数据要尽量一致，否则奖励模型会学到混乱信号。

标注标准需要明确：

- 准确性优先还是简洁性优先？
- 遇到不确定事实要不要拒答？
- 是否必须解释推理过程？
- 安全问题如何处理？
- 引用错误是否一票否决？
- 长回答和短回答如何比较？

如果标注者 A 喜欢详细，标注者 B 喜欢简短，奖励模型会学到摇摆偏好。

所以 RLHF 不是单纯算法问题，也是数据规范问题。

## 11. 奖励模型也会错

奖励模型只是另一个模型。

它可能：

- 偏爱长回答。
- 偏爱固定格式。
- 误判专业问题。
- 被表面流畅度欺骗。
- 对少数语言或领域效果差。
- 无法可靠判断复杂事实。

因此 PPO 优化出来的模型，不一定在所有维度都更好。

训练后必须做独立评测，而不是只看 reward 分数。

这和 RAG 评测类似：优化指标不等于用户价值。

## 12. RLHF 和应用层工程的关系

RLHF 改善的是基础模型行为，但应用层仍然需要工程控制。

例如：

| 问题 | RLHF 能否单独解决 |
| --- | --- |
| 私有知识更新 | 不能，需要 RAG |
| 权限过滤 | 不能，需要工具层 |
| 引用真实性 | 不能，需要证据校验 |
| API 写操作安全 | 不能，需要确认点 |
| 长任务执行 | 不能，需要工作流/agent |
| 线上质量回归 | 不能，需要评测集 |

所以不要把 RLHF 当成应用系统的替代品。

更合理的关系是：

```text
RLHF 让模型更会当助手
RAG 让模型拿到外部证据
工具调用让模型执行动作
上下文工程让信息组织可控
评测让系统持续变好
```

## 13. 小结

RLHF 主线可以用一句话概括：

```text
先用 SFT 教模型如何回答，再用人类偏好训练奖励模型，最后用 PPO 让模型更倾向于生成高偏好回答。
```

其中：

- SFT 让模型进入助手模式。
- 偏好数据表达人类更喜欢什么。
- 奖励模型把偏好变成分数。
- PPO 用分数优化模型。
- KL 约束防止模型偏离太远。
