RAG 经常被一句话概括成“先检索，再让大模型回答”。

这句话没错，但太粗了。真正做系统时，问题通常不在“能不能搜到一点内容”，而在这些细节：

- 文档应该按整篇存，还是切成段落？
- 用户问一句话，为什么可以召回相关资料？
- 检索结果怎么塞进 prompt，才不会浪费上下文？
- 模型回答里的引用怎么保证真的来自原文？
- 搜不到证据时，应该让模型猜，还是明确说不知道？

RAG 的核心不是把数据库直接丢给大模型。它更像是在回答之前，先临时组装一份“可追溯的证据包”，让模型只能基于这份证据回答。

这篇文章接着上一篇 tokenizer 和上下文成本，讲 RAG 的入门链路：文档切分、向量召回、上下文组装，以及带引用回答。

## 1. RAG 解决的不是“模型不够聪明”

很多人第一次做 RAG，是因为发现模型答不出自己的业务资料。

例如：

- 公司内部制度没有出现在模型训练数据里。
- 产品说明刚更新，模型不知道最新版本。
- 客服知识库有权限隔离，不能全部暴露给所有用户。
- 法务、财务、技术文档需要精确引用，不能只给一个看起来合理的回答。

这些问题不适合靠“再训练一个模型”直接解决。训练成本高，更新慢，权限难控制，也不容易追溯答案来源。

RAG 更适合处理这类场景：

```text
用户问题
  -> 检索相关文档片段
  -> 把片段作为上下文交给大模型
  -> 大模型基于片段回答
  -> 回答中附上来源引用
```

所以 RAG 解决的是知识更新、私有知识、权限过滤和答案可追溯的问题，而不是单纯让模型“更聪明”。

## 2. 最小可用 RAG 链路

一个最小可用的 RAG 系统可以拆成两条链路。

第一条是离线或异步的入库链路：

```text
原始文档
  -> 解析
  -> 清洗
  -> 切分 chunk
  -> 生成 embedding
  -> 写入检索索引
```

第二条是用户提问时的在线链路：

```text
用户问题
  -> 生成 query embedding
  -> 向量召回 topK chunks
  -> 权限过滤和去重
  -> 组装证据上下文
  -> LLM 生成带引用回答
```

这两条链路要分开理解。入库链路关心“资料怎么变成可检索的知识块”，在线链路关心“问题怎么找到证据，并让模型稳定回答”。

很多 RAG 项目不稳定，是因为只做了“把文本塞进向量库”和“把搜索结果塞进 prompt”，中间缺少清洗、chunk、过滤、引用、评测这些工程环节。

## 3. 文档和 chunk 不是一回事

RAG 里最重要的建模区别是：原始文档和检索 chunk 不是同一层对象。

一篇原始文档可能是：

- 一篇产品说明。
- 一份 PDF 合同。
- 一个知识库页面。
- 一篇 Markdown 文档。
- 一条从 PostgreSQL 里读取出来的业务文章。

但检索时真正被召回的，通常不是整篇文档，而是文档里的某个片段。

可以把数据模型简化成这样：

```ts
type Document = {
  id: string
  title: string
  sourceUrl?: string
  content: string
  updatedAt: string
}

type Chunk = {
  id: string
  documentId: string
  title: string
  sectionPath: string[]
  content: string
  chunkIndex: number
  embedding: number[]
}
```

`Document` 用来保存原文和业务元数据，`Chunk` 用来检索。

如果直接把整篇文档作为一条检索记录，会遇到两个问题：

- 召回粒度太粗。用户只问退款条件，系统却把整篇售后政策塞给模型。
- 上下文浪费严重。模型读到大量无关内容，既贵又容易分心。

所以 RAG 通常要先切 chunk。

## 4. 入库前先清洗文档

切 chunk 之前要先清洗文档。否则检索索引里会混进很多无意义内容。

网页内容里常见噪声包括：

- 导航栏、页脚、版权声明。
- 相关推荐、广告、侧边栏。
- 重复标题和面包屑。
- 分享按钮、登录提示、无关脚本文本。

业务文档里也会有噪声：

- 模板占位符。
- 导出工具生成的页码和页眉。
- 表格里重复出现的列名。
- 已废弃但没有标记的旧版本内容。

清洗的目标不是把文本变短，而是保留可回答问题的正文和必要上下文。

一个比较稳妥的清洗结果应该包含：

```json
{
  "documentId": "refund_policy_2026",
  "title": "退款规则说明",
  "sourceUrl": "https://example.com/docs/refund",
  "updatedAt": "2026-06-12T10:00:00+08:00",
  "content": "用户在支付成功后 7 天内可以申请退款..."
}
```

标题、来源、更新时间这些元数据不要丢。它们后面会用于过滤、排序和引用。

## 5. chunk 切分为什么重要

chunk 是 RAG 的基本检索单位。

chunk 太小，会丢上下文。例如只切到一句话：

```text
支持 7 天内退款。
```

这句话本身不完整。它没有说明适用范围、例外条件、是否包含虚拟商品。

chunk 太大，又会带入太多无关信息。例如把整章售后政策放进一个 chunk，用户问“虚拟商品能退吗”，模型可能要在几千字里找答案。

比较常见的做法是按结构优先切分：

```text
文档标题
  -> 一级标题
  -> 二级标题
  -> 段落
  -> 必要时按 token 长度继续拆
```

可以给每个 chunk 保留一小段重叠内容，避免答案刚好跨在两个 chunk 边界上：

```text
chunk 1: A B C D
chunk 2: D E F G
chunk 3: G H I J
```

重叠不是越多越好。重叠太多会让索引膨胀，也会导致召回结果重复。

实际工程里可以从一个保守配置开始：

| 参数 | 起步建议 | 影响 |
| --- | --- | --- |
| chunk 大小 | 300 到 800 token | 越大越保留上下文，越小越精准 |
| overlap | 50 到 120 token | 减少边界丢失，但会增加重复 |
| topK | 5 到 10 | 召回更多证据，但会占上下文 |
| 最低分数 | 按数据集调 | 防止无关内容进入回答 |

这不是固定标准。技术文档、客服知识库、法律合同、表格数据的最佳切分方式都不一样。

## 6. embedding 和向量召回在做什么

embedding 可以把一段文本变成一组数字。

非常简化地看：

```text
"如何申请退款？"
  -> embedding model
  -> [0.12, -0.08, 0.31, ...]
```

语义相近的文本，向量距离通常更近。用户问“怎么退钱”，文档里写的是“退款申请流程”，即使关键词不完全一样，也可能被召回。

向量召回的直觉是：

```text
把用户问题变成向量
把每个 chunk 也变成向量
找出距离最近的几个 chunk
```

常见相似度指标包括 cosine similarity、dot product 和 L2 distance。具体用哪一种，通常取决于 embedding 模型和向量数据库的建议配置。

但要注意：embedding 不是事实判断器。

它只能告诉你“这两个文本在语义空间里可能接近”，不能保证召回结果一定正确，也不能保证结果足够回答用户问题。因此 RAG 不能只依赖向量召回，还需要过滤、去重、排序和引用校验。

## 7. 一次查询应该怎么处理

用户提问时，可以按下面的顺序处理：

```text
1. 标准化问题
2. 生成 query embedding
3. 按权限和业务范围过滤候选文档
4. 向量召回 topK chunks
5. 去除重复或过短片段
6. 判断分数是否足够
7. 组装证据上下文
8. 调用 LLM 生成回答
```

权限过滤非常重要。

如果系统里有多租户、用户组、付费等级或内部资料，不能先全库召回再靠 prompt 告诉模型“不要使用无权限内容”。权限应该在检索阶段就生效。

例如：

```ts
type SearchFilters = {
  tenantId: string
  userGroupIds: string[]
  sourceTypes: string[]
  publishedOnly: boolean
}
```

检索服务应该只返回当前用户有权看的 chunks。

还要处理低置信度结果。如果 topK 里最高分都很低，说明知识库可能没有答案。这时候正确行为不是让模型猜，而是返回：

```text
我没有在当前知识库中找到足够证据回答这个问题。
```

这类“拒答”能力，是 RAG 系统可信度的一部分。

## 8. 上下文组装比召回更容易被低估

召回到 chunks 后，不应该直接把 JSON 原样塞进 prompt。

更好的做法是把证据整理成模型容易使用的格式：

```text
[1] 退款规则说明 / 虚拟商品
来源：https://example.com/docs/refund
更新时间：2026-06-12
内容：
虚拟商品一经兑换不支持退款；未兑换的虚拟商品可在支付后 7 天内申请退款。

[2] 退款规则说明 / 退款流程
来源：https://example.com/docs/refund
更新时间：2026-06-12
内容：
用户可以在订单详情页提交退款申请，系统会在 3 个工作日内完成审核。
```

这里的 `[1]`、`[2]` 就是后面回答里的引用编号。

上下文组装时要注意几件事：

- 把最相关的证据放前面。
- 合并同一文档里相邻或重复的 chunk。
- 保留标题、章节、来源和更新时间。
- 控制总 token 数，不要超过模型上下文预算。
- 不要把检索分数、内部 ID、权限字段全部暴露给模型。

RAG 不是召回越多越好。召回太多，模型会在一堆弱相关证据里迷路。

## 9. 带引用回答的 prompt

带引用回答的关键，是让模型只能使用证据区里的内容。

一个简化 prompt 可以这样写：

```text
你是知识库问答助手。

规则：
1. 只能基于「证据」回答。
2. 如果证据不足，直接说明无法从当前知识库确认。
3. 每个关键结论后面都要标注引用编号，例如 [1]。
4. 不要编造证据编号，不要引用证据中不存在的信息。

用户问题：
{question}

证据：
{evidence_blocks}

请给出简洁、准确、带引用的回答。
```

这个 prompt 的重点不是让模型“变诚实”，而是给输出建立约束。

如果没有证据，回答应该是：

```text
当前知识库里没有足够证据确认这个问题。
```

而不是：

```text
根据一般经验，可能是...
```

在业务系统里，宁可少答，也不要编一个看起来合理但无法追溯的答案。

## 10. 引用不是装饰

很多 RAG demo 会在回答末尾放几个来源链接，但这还不够。

真正有用的引用应该满足三个条件：

- 回答中的关键结论能对应到具体 chunk。
- 引用编号确实存在于证据列表里。
- 用户点开来源后，能看到原文或对应章节。

例如：

```text
未兑换的虚拟商品可以在支付后 7 天内申请退款；已兑换的虚拟商品不支持退款 [1]。退款申请需要在订单详情页提交，审核通常在 3 个工作日内完成 [2]。
```

这比只在末尾写“参考：退款规则说明”更可靠。

系统层面最好做一次引用校验：

```ts
function validateCitations(answer: string, evidenceIds: string[]) {
  const citedIds = [...answer.matchAll(/\[(\d+)\]/g)].map((match) => match[1])
  return citedIds.every((id) => evidenceIds.includes(id))
}
```

这只能检查“编号有没有乱编”，不能检查“引用内容是否真的支持结论”。更严格的系统还会做二次校验，例如让另一个模型判断每条结论是否被引用证据支持。

## 11. 常见失败模式

RAG 最常见的问题不是“完全搜不到”，而是“搜到了但不够准”。

常见失败模式包括：

| 失败模式 | 表现 | 处理方式 |
| --- | --- | --- |
| chunk 太粗 | 回答包含无关内容 | 缩小 chunk，按标题结构切 |
| chunk 太碎 | 召回片段缺少上下文 | 增加 overlap，合并相邻 chunk |
| 噪声太多 | 召回导航、页脚、模板文字 | 入库前清洗正文 |
| 只做向量召回 | 专有名词、编号、代码搜不到 | 增加关键词或混合检索 |
| 没有权限过滤 | 用户看到不该看的内容 | 检索阶段加 tenant 和 ACL |
| 没有拒答 | 模型在证据不足时猜测 | 设置最低分数和拒答规则 |
| 引用不校验 | 回答引用不存在的编号 | 输出后做 citation validation |

这些问题都不是单靠换一个更大的模型就能解决的。

RAG 是一个信息工程问题，不只是一个 prompt 问题。

## 12. 怎么评估一个 RAG 系统

没有评测集的 RAG 很难持续优化。

可以从小评测集开始，先准备 20 到 50 个真实问题。每个问题至少记录：

```json
{
  "question": "虚拟商品兑换后还能退款吗？",
  "expectedSources": ["refund_policy_2026#virtual-goods"],
  "expectedAnswerPoints": [
    "已兑换虚拟商品不支持退款",
    "未兑换虚拟商品可在 7 天内申请退款"
  ]
}
```

然后分别评估检索和回答。

检索侧可以看：

- `recall@k`：正确证据是否出现在前 k 个结果里。
- 重复率：topK 是否被同一段相似内容占满。
- 无关率：召回结果里有多少片段明显无关。

回答侧可以看：

- 是否基于证据回答。
- 是否覆盖关键结论。
- 是否引用正确。
- 证据不足时是否拒答。
- 有没有把旧版本内容当成新版本。

这套评测不需要一开始很复杂，但必须存在。否则每次调 chunk 大小、topK、embedding 模型、prompt，都只能凭感觉判断。

## 13. 一个最小 TypeScript 示例

下面是一个简化版 RAG 查询流程。真实系统会把 embedding、向量库、权限、日志和模型调用拆成独立服务，这里只展示主干。

```ts
type RetrievedChunk = {
  id: string
  title: string
  sectionPath: string[]
  sourceUrl?: string
  updatedAt: string
  content: string
  score: number
}

async function answerWithRag(question: string, userId: string) {
  const filters = await getSearchFiltersForUser(userId)
  const queryEmbedding = await embed(question)

  const chunks = await vectorSearch({
    embedding: queryEmbedding,
    topK: 8,
    filters
  })

  const usefulChunks = dedupeAndFilter(chunks, {
    minScore: 0.72,
    maxChunks: 5
  })

  if (usefulChunks.length === 0) {
    return {
      answer: "当前知识库里没有足够证据回答这个问题。",
      citations: []
    }
  }

  const evidence = usefulChunks.map((chunk, index) => ({
    citationId: String(index + 1),
    chunk
  }))

  const prompt = buildRagPrompt({
    question,
    evidence
  })

  const answer = await generateAnswer(prompt)
  const valid = validateCitations(
    answer,
    evidence.map((item) => item.citationId)
  )

  return {
    answer: valid ? answer : "回答引用校验失败，请重新检索或缩小问题范围。",
    citations: evidence.map((item) => ({
      id: item.citationId,
      title: item.chunk.title,
      sectionPath: item.chunk.sectionPath,
      sourceUrl: item.chunk.sourceUrl,
      updatedAt: item.chunk.updatedAt
    }))
  }
}
```

这个例子里最重要的不是代码本身，而是几个边界：

- 检索前先拿用户权限过滤条件。
- 低分结果直接拒答。
- 证据块有稳定的 citation id。
- 生成后做引用编号校验。
- 返回给前端的不只是 answer，还有 citations。

这样前端才能在 blog 页面或问答入口里展示“回答 + 来源”。

## 14. LLMWiki、博客和 agent 的关系

如果后续要在 blog 页面放一个问答入口，建议把三层拆开：

```text
博客页面
  -> 展示文章、问答入口、引用来源

LLMWiki / 知识层
  -> 保存适合模型读取的 Markdown、章节、引用和索引信息

Agent / 问答服务
  -> 调用搜索工具、读取工具、LLM，生成带引用回答
```

博客页面不一定要知道 Meilisearch、向量库或 embedding 的细节。它只需要调用一个问答 API：

```http
POST /api/ask
```

请求：

```json
{
  "question": "RAG 为什么要切 chunk？",
  "scope": "blog"
}
```

响应：

```json
{
  "answer": "RAG 切 chunk 是为了让召回粒度更接近问题，并减少无关上下文进入模型 [1]。",
  "citations": [
    {
      "id": "1",
      "title": "RAG 入门：文档切分、向量召回和带引用回答",
      "url": "/blog/rag-introduction-retrieval-citations"
    }
  ]
}
```

LLMWiki 更适合承担“内容被模型读取”的那一层：结构化 Markdown、章节关系、来源链接、知识图谱或索引元数据。Agent 则负责把这些能力编排成一次问答。

也就是说，网站是一套用户体验，agent 可以是一套独立服务。两者通过 API 连接。这样以后你要把检索范围从 blog 扩展到 PostgreSQL 里的业务知识库，也不用重写前端。

## 15. 入门检查清单

如果要判断一个 RAG demo 是否已经接近可用，可以看下面这张清单：

- 文档入库前是否清洗过正文？
- chunk 是否保留标题、章节、来源和更新时间？
- 检索是否支持权限过滤？
- 是否设置了最低召回分数？
- 是否能在证据不足时拒答？
- prompt 是否要求只能基于证据回答？
- 回答是否带具体引用编号？
- 引用编号是否做了校验？
- 前端是否能展示引用来源？
- 是否有一小批真实问题用于评测？

这些都做到后，RAG 才不只是一个 demo，而是一个可以继续演进的知识问答系统。

后续再往前走，就会进入 RAG 进阶：混合检索、重排、GraphRAG 和评测集。那部分解决的是“向量召回不够准”之后的问题。
