上一篇讲了经典 RLHF 主线：

```text
SFT
  -> 偏好数据
  -> 奖励模型
  -> PPO
  -> 对齐模型
```

这条路线有效，但工程复杂：要训练奖励模型，要跑 PPO，要控制 KL，要处理 reward hacking。

后来出现了很多更直接的偏好优化方法，比如 DPO。推理模型训练里也常见 GRPO 这类方法，用来减少传统 PPO 的复杂度。

这一篇不追公式细节，而是讲清核心差异：

- DPO 为什么能绕开显式奖励模型。
- 它和 RLHF/PPO 的关系是什么。
- GRPO 这类方法为什么适合推理训练场景。
- 偏好优化和推理强化学习到底关注什么。

## 1. 先回顾 RLHF 的复杂点

经典 RLHF 里有几个组件：

```text
policy model：正在优化的模型
reference model：参考模型，用于 KL 约束
reward model：学习人类偏好的打分器
PPO：用 reward 更新 policy
```

这套系统复杂在：

- 奖励模型要单独训练。
- PPO 训练不稳定，需要调参。
- 采样、打分、更新流程重。
- reward model 可能被模型钻漏洞。
- KL 系数、优势估计、batch 设置都会影响结果。

如果目标只是让模型更偏向 chosen、远离 rejected，有没有办法更直接？

DPO 就是这个方向。

## 2. DPO 的直觉

DPO 是 Direct Preference Optimization。

它直接使用偏好对：

```json
{
  "prompt": "解释 RAG。",
  "chosen": "RAG 是检索增强生成...",
  "rejected": "RAG 是一种数据库..."
}
```

目标是让模型对 `chosen` 的概率相对更高，对 `rejected` 的概率相对更低。

直觉上：

```text
同一个 prompt 下
  提高 chosen 的相对概率
  降低 rejected 的相对概率
  同时不要偏离参考模型太远
```

它不需要显式训练一个奖励模型，也不需要跑完整 PPO。

这让偏好优化流程更简单。

## 3. DPO 和 SFT 的区别

SFT 只看正样本：

```text
prompt -> chosen
```

训练目标是让模型更容易生成 chosen。

DPO 同时看正负样本：

```text
prompt -> chosen better than rejected
```

它学习的是偏好差异。

例如：

```text
chosen：简洁解释 RAG，并说明引用价值。
rejected：只说 RAG 是搜索加生成，没有提引用。
```

SFT 只知道 chosen 好。DPO 知道 chosen 比 rejected 好。

这让模型更容易学习细微偏好。

## 4. DPO 和奖励模型的关系

DPO 并不是说“没有奖励”。

更准确地说，它把奖励建模和策略优化合在了一个目标里，不再显式训练一个单独 reward model。

经典 RLHF：

```text
偏好数据 -> reward model
policy 生成回答 -> reward model 打分 -> PPO 更新 policy
```

DPO：

```text
偏好数据 -> 直接更新 policy
```

这让工程链路更短，也更容易复现。

但 DPO 仍然依赖高质量偏好数据。如果 chosen/rejected 标注不好，模型一样会学坏。

## 5. DPO 的优势

DPO 的主要优势：

- 不需要单独训练奖励模型。
- 不需要复杂 PPO rollout。
- 训练过程更接近普通监督训练。
- 实现和调参相对简单。
- 适合已有偏好数据的场景。

对于很多指令对齐任务，DPO 是很实用的选择。

例如：

- 让回答更简洁。
- 让模型更遵循格式。
- 减少胡乱引用。
- 学习某种产品语气。
- 改善拒答边界。

只要能构造 chosen/rejected，就可以做偏好优化。

## 6. DPO 的局限

DPO 也不是万能。

第一，它依赖偏好对。

如果问题没有高质量 rejected，训练信号会弱。

第二，它更适合离线偏好数据。

如果需要模型在线探索、尝试不同解法、根据环境反馈学习，DPO 就不一定够。

第三，它不自动解决可验证推理。

数学题、代码题、复杂工具任务，偏好不只是“哪个回答看起来更好”，而是结果是否正确。

这就进入推理训练和强化学习场景。

## 7. 推理训练关注什么

推理类任务的特点是：结果往往可以验证。

例如：

- 数学题答案是否正确。
- 代码是否通过测试。
- SQL 查询结果是否符合预期。
- Agent 是否完成任务。
- 工具调用轨迹是否满足约束。

这类任务不一定需要人工偏好标注每个回答。可以用验证器给反馈：

```text
生成多个解法
  -> 执行或验证
  -> 正确结果给高奖励
  -> 错误结果给低奖励
```

这和普通偏好优化不同。它更像让模型通过可验证结果学习。

## 8. GRPO 的直觉

GRPO 可以理解为一种更轻量的强化学习思路，常被放在推理训练语境里讨论。

它的核心直觉是：对同一个问题采样一组回答，在组内比较好坏，用相对优势更新模型。

简化流程：

```text
给定一个 prompt
  -> 模型采样多个回答
  -> 用规则或验证器打分
  -> 计算组内相对优势
  -> 更新模型
```

和 PPO 相比，这类方法通常试图减少对单独价值模型的依赖，让训练流程更轻。

关键点不在名字，而在训练信号：

```text
同一问题的多个候选之间，哪个结果更好？
```

这和推理任务天然匹配。

## 9. 偏好优化和推理 RL 的差异

可以这样区分：

| 维度 | 偏好优化 | 推理强化学习 |
| --- | --- | --- |
| 数据来源 | 人类偏好对 | 采样结果 + 验证器 |
| 目标 | 更符合人类偏好 | 更高可验证正确率 |
| 反馈 | chosen/rejected | reward 或 pass/fail |
| 典型任务 | 对话、风格、安全 | 数学、代码、工具任务 |
| 关键风险 | 标注不一致 | 奖励稀疏、过拟合验证器 |

两者可以结合。

例如一个代码助手：

- 用 DPO 学习更好的回答风格。
- 用测试通过率训练代码生成。
- 用安全偏好数据减少危险建议。

## 10. 为什么推理训练要多采样

推理任务里，模型一次生成可能错，但多采样可能有正确答案。

例如同一道题采样 8 个解法：

```text
answer 1: wrong
answer 2: wrong
answer 3: correct
answer 4: wrong
answer 5: correct
...
```

如果验证器能识别正确答案，就可以用这些样本构造训练信号。

这也是为什么推理模型训练经常关注：

- rollout 数量。
- 采样温度。
- 奖励设计。
- 验证器质量。
- 长链路 credit assignment。

模型不是只学“答案是什么”，还学“哪些生成轨迹更可能到达正确答案”。

## 11. 奖励设计仍然是核心

无论 DPO、PPO 还是 GRPO，都绕不开训练信号。

对于推理任务，奖励可以来自：

- 单元测试是否通过。
- 数学答案是否匹配。
- 格式是否合规。
- 工具调用是否成功。
- 是否触发不该触发的动作。
- 是否在需要确认时停下。

奖励设计不好，模型会学会投机。

例如代码任务只奖励“输出看起来像代码”，模型就会生成漂亮但不能运行的代码。

如果只奖励“通过样例测试”，模型可能过拟合样例。

奖励要尽可能贴近真实目标。

## 12. 和 Agent 训练的关系

Agent 训练比普通推理训练更复杂。

因为 agent 的输出不是一段文本，而是一条轨迹：

```text
观察
  -> 调用搜索
  -> 读取文件
  -> 修改代码
  -> 运行测试
  -> 修复错误
  -> 完成任务
```

评估 agent 不能只看最终回答，还要看：

- 是否调用正确工具。
- 是否越权。
- 是否重复调用。
- 是否正确处理错误。
- 是否产生真实可验证结果。
- 是否在高风险动作前确认。

因此 agent 训练需要 rollout、沙箱、评测和轨迹日志。

## 13. 小结

DPO 和 GRPO 代表了后训练中的两个重要方向：

- DPO：更直接地用偏好对优化模型，不显式训练奖励模型和 PPO。
- GRPO 这类方法：在推理场景里通过一组候选和可验证奖励优化模型。

它们和经典 RLHF 的关系不是简单替代，而是面向不同成本和任务形态的工程选择。

如果任务主要是对话偏好，DPO 很实用。

如果任务有可验证结果，推理强化学习更重要。

如果任务是 agent，训练对象就从单个回答扩展成完整执行轨迹。
