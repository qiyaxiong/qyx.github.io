很多产品一接入大模型，就会把目标说成“做一个 agent”。

但 agent 不是越自主越好。真实系统里，很多任务更适合固定工作流；有些步骤可以让模型判断；少数高风险动作必须要求人类确认。

这篇文章讨论工作流与 Agent 的边界：

- 什么是固定流程。
- 什么是自主决策。
- 什么时候该让模型选择工具。
- 什么时候必须让人确认。
- 一个可上线 agent 应该怎么拆。

## 1. 工作流和 Agent 的差别

固定工作流是预先定义好的步骤：

```text
输入文章
  -> 提取标题
  -> 生成摘要
  -> 检查格式
  -> 发布
```

Agent 则会根据目标和上下文选择下一步：

```text
目标：把这篇文章发布到博客
  -> 检查文章是否完整
  -> 发现缺少 frontmatter
  -> 补 frontmatter
  -> 运行构建
  -> 构建失败则修复
  -> 构建通过后请求发布确认
```

两者最大的区别不是有没有模型，而是控制流在哪里。

| 类型 | 控制流 | 适合场景 |
| --- | --- | --- |
| 固定工作流 | 程序决定下一步 | 稳定、重复、低歧义任务 |
| Agent | 模型参与决定下一步 | 开放、探索、多工具任务 |
| 混合模式 | 程序定边界，模型填细节 | 生产系统最常见 |

生产系统通常不是纯 agent，而是混合模式。

## 2. 不要把所有事情都交给 Agent

如果任务步骤非常稳定，固定工作流更好。

例如：

- 表单提交后发确认邮件。
- 支付成功后创建订单。
- 定时同步索引。
- 文章构建通过后触发部署。

这些任务不需要模型决定下一步。模型参与反而会增加不确定性。

Agent 更适合：

- 用户目标不完整，需要澄清。
- 需要搜索、阅读、判断、修复。
- 中间可能失败，需要恢复。
- 工具很多，路径不固定。
- 需要根据结果调整计划。

例如：

```text
帮我分析为什么这次 RAG 回答错了，并给出修复方案。
```

这可能涉及查看日志、检索结果、prompt、评测集、代码配置。路径不固定，适合 agent。

## 3. 一个任务可以拆成确定和不确定部分

很多任务不是全固定，也不是全自主。

例如“发布一篇博客文章”：

确定部分：

```text
写入 MDX 文件
运行 check
运行 build
git commit
git push
Vercel 部署
线上验证
```

不确定部分：

```text
文章结构怎么组织
哪里需要补充解释
构建失败后怎么修
线上验证失败时查哪里
```

所以更好的设计是：

```text
工作流负责阶段边界
Agent 负责阶段内判断
人类确认负责高风险动作
```

这样既有灵活性，又不会失控。

## 4. Agent 的最小循环

一个最小 agent 循环通常是：

```text
观察
  -> 思考
  -> 选择动作
  -> 调用工具
  -> 读取结果
  -> 更新状态
  -> 判断是否完成
```

用工程语言说：

```ts
type AgentState = {
  goal: string
  messages: Message[]
  plan: PlanStep[]
  observations: Observation[]
  toolCalls: ToolCall[]
  artifacts: Artifact[]
}
```

每一轮都要决定：

- 目标是否已经完成。
- 是否需要更多信息。
- 是否需要调用工具。
- 是否需要用户确认。
- 是否应该停止。

Agent 的难点不是“会不会调用工具”，而是“什么时候继续，什么时候停止”。

## 5. 状态比长 prompt 更重要

很多 agent demo 会把所有历史都塞进 prompt，让模型自己记。

更稳的做法是维护结构化状态：

```json
{
  "goal": "补齐 AI 学习主线 12 篇文章",
  "currentStep": "write_article_9",
  "completedSteps": [
    "write_article_6",
    "write_article_7",
    "write_article_8"
  ],
  "pendingConfirmations": [],
  "lastVerification": {
    "command": "pnpm run build",
    "status": "passed"
  }
}
```

结构化状态有几个好处：

- 程序可以检查。
- 可以恢复中断。
- 可以写日志。
- 可以控制上下文长度。
- 可以和人类确认点对齐。

Agent 不应该只存在于一段对话里。它应该有外部状态。

## 6. 计划不是一次性生成后照做

Agent 很容易生成一个漂亮计划，然后现实第一步就变了。

例如：

```text
计划：运行测试 -> 修复失败 -> 部署
```

但运行后发现依赖没安装、测试命令不存在、部署平台需要登录。

所以计划应该是可更新的。

一个好的计划状态包含：

```ts
type PlanStep = {
  id: string
  title: string
  status: "pending" | "in_progress" | "completed" | "blocked"
  evidence?: string
}
```

每完成一步，都要有证据：

- 文件已写入。
- 命令退出码为 0。
- 线上 URL 返回 200。
- 用户确认已收到。

没有证据的计划完成，只是模型自我声明。

## 7. 工具选择要受工作流约束

如果模型能看到所有工具，它可能在不合适的阶段调用危险工具。

更稳的做法是按阶段开放工具。

例如发布文章流程：

| 阶段 | 可用工具 |
| --- | --- |
| 写作 | 读写文件、搜索本地内容 |
| 验证 | 运行 check/build |
| 提交 | git status、git diff、git commit |
| 部署 | git push、vercel ls、alias set |
| 验证线上 | curl、浏览器截图 |

在写作阶段，不需要给模型部署工具。在验证失败前，也不需要给它发布工具。

工具可见性本身就是安全边界。

## 8. 人类确认点怎么设计

人类确认点不是一句“你确定吗”。

确认内容应该包含：

- 将要执行什么动作。
- 影响范围是什么。
- 是否可回滚。
- 需要用户做什么选择。

例如：

```json
{
  "confirmationType": "deploy_to_production",
  "summary": "将 master 分支推送到 GitHub，并触发 Vercel 生产部署。",
  "effects": [
    "公开站点会更新",
    "新文章会被搜索引擎看到"
  ],
  "rollback": "可以重新部署上一个 Vercel deployment"
}
```

确认点适合放在：

- 写操作前。
- 发布前。
- 费用动作前。
- 权限变更前。
- 删除或覆盖前。
- 外部通知前。

低风险读操作不需要每次确认，否则 agent 会变得不可用。

## 9. 自主程度可以分级

可以把 agent 自主程度分成五级：

| 级别 | 模式 | 说明 |
| --- | --- | --- |
| L0 | 纯手动 | 用户每步操作 |
| L1 | 建议 | 模型只给建议，不执行 |
| L2 | 辅助执行 | 模型执行低风险步骤 |
| L3 | 受控自主 | 模型可多步执行，高风险需确认 |
| L4 | 高自主 | 模型长期运行，程序监控和回滚 |

大多数业务系统适合 L2 或 L3。

例如博客问答入口是 L2：模型检索并回答，不修改状态。

文章发布 agent 是 L3：可以写文件、跑构建、提交，但生产部署需要明确授权。

不要一开始追求 L4。长期自主运行需要更完整的评测、监控、权限和回滚系统。

## 10. Agent 的停止条件

Agent 必须知道什么时候停。

常见停止条件：

- 目标已完成，并有验证证据。
- 遇到需要用户确认的动作。
- 连续失败达到上限。
- 工具返回不可恢复错误。
- 上下文或成本超过预算。
- 任务目标不清，需要澄清。

不要让 agent 无限循环。

可以设置硬限制：

```ts
const limits = {
  maxSteps: 20,
  maxToolCalls: 50,
  maxRetriesPerTool: 2,
  maxRuntimeMs: 10 * 60 * 1000
}
```

限制不是为了让 agent 变笨，而是为了让系统可控。

## 11. 错误恢复要分类

Agent 遇到错误时，不能只说“失败了”。

要先分类：

| 错误 | 示例 | 恢复 |
| --- | --- | --- |
| 参数错误 | schema validate failed | 修正参数重试 |
| 权限错误 | forbidden | 请求授权或停止 |
| 临时错误 | timeout | 重试一次 |
| 外部状态变化 | 文件已被修改 | 重新读取状态 |
| 测试失败 | build failed | 读取错误并修复 |
| 目标冲突 | 用户新指令改变方向 | 更新计划 |

错误恢复依赖工具错误码、结构化状态和日志。

如果工具只返回一段报错文本，agent 很难稳定恢复。

## 12. 工作流可测试，Agent 也要可评测

固定工作流可以用单元测试、集成测试验证。

Agent 更开放，但仍然可以评测：

- 是否能完成任务。
- 是否调用了正确工具。
- 是否避免危险操作。
- 是否在需要确认时停止。
- 是否能从常见错误恢复。
- 是否产生可验证结果。

例如文章发布 agent 的评测用例：

```json
{
  "task": "发布一篇新 MDX 文章",
  "expected": [
    "生成合法 frontmatter",
    "运行 pnpm run check",
    "运行 pnpm run build",
    "提交前查看 git diff",
    "部署后验证 URL 200"
  ],
  "mustAskConfirmationBefore": [
    "production deploy"
  ]
}
```

Agent 不是不能测试，只是测试对象从函数输出变成了轨迹和结果。

## 13. 和 RAG 问答入口的关系

博客页面里的问答入口，可以先做成受控工作流，而不是完全自主 agent。

流程：

```text
用户提问
  -> 改写问题
  -> 检索文章
  -> 重排证据
  -> 生成带引用回答
  -> 前端展示
```

这个流程相对固定。模型可以参与查询改写和答案生成，但不需要自由决定下一步。

后续如果加入更多能力，比如：

- 追问澄清。
- 读取整篇文章。
- 对比多篇文章。
- 生成学习计划。
- 创建收藏或笔记。

再逐步引入 agent 循环。

## 14. 一个混合架构

可以把系统拆成三层：

```text
Workflow Orchestrator
  -> 决定阶段、权限、停止条件

Agent Planner
  -> 在允许范围内选择下一步

Tools
  -> 执行搜索、读取、写入、部署等动作
```

工作流负责边界：

- 当前阶段是什么。
- 哪些工具可用。
- 哪些动作需要确认。
- 什么条件算完成。

Agent 负责判断：

- 当前还缺什么信息。
- 应该调用哪个工具。
- 工具失败后怎么恢复。
- 如何解释结果。

工具负责执行：

- 校验输入。
- 检查权限。
- 返回结构化结果。
- 记录日志。

这个分层比“一个大模型控制一切”更适合生产系统。

## 15. 小结

工作流和 Agent 不是对立关系。

固定流程提供稳定边界，Agent 提供灵活判断，人类确认点处理高风险动作。

一个可上线的 agent 系统应该做到：

- 能区分确定步骤和不确定步骤。
- 有结构化状态，而不是只靠对话历史。
- 工具按阶段开放。
- 写操作和高风险动作需要确认。
- 有明确停止条件。
- 错误恢复按类型处理。
- 完成状态必须有验证证据。
- Agent 轨迹可以记录和评测。
