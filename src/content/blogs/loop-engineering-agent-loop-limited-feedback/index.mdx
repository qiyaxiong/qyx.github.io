最近看了 B 站视频 [《Loop工程：Agent Loop到底 是什么？》](https://www.bilibili.com/video/BV1cVjN6oEwx/)。

这条视频最有价值的地方，不是又发明了一个新名词，而是把 Agent Loop 的边界讲得很直接：全自动循环听起来很酷，但在真实开发里很容易变成代币黑洞、错误假设放大器和质量不可控的自动化幻觉。

所以这篇不是复述视频，而是整理一份实践笔记：

- 什么是 Agent Loop。
- 为什么全自动 loop 容易失控。
- 什么样的 loop 真的有用。
- 它和前面写过的 Prompt Engineer、Content Engineer、Harness Engineer 有什么区别。
- 如果要把它变成工程能力，Loop Engineer 应该关心什么。

## 1. Agent Loop 不是让 AI 一直自己跑

很多人听到 Agent Loop，会自然想到这样的流程：

```text
给 AI 一个目标
  -> AI 写代码
  -> AI 自己检查
  -> AI 自己修复
  -> AI 自己继续
  -> 直到完成
```

这就是最有诱惑力、也最危险的理解。

它的问题在于：循环本身不等于进步。一个系统会反复行动，不代表它每一轮都在接近目标。

如果目标不清楚、反馈不可靠、停止条件不存在，loop 只会把错误放大：

- 一开始误解需求，后面每一轮都在错误方向上精修。
- 代码能跑，但产品体验越来越偏。
- 测试通过，但用户真正关心的细节没有被检查。
- 模型不断修改，diff 越来越大，人类越来越难接管。
- 每一轮都消耗 token，但没有产生可验证的增量质量。

所以 Agent Loop 的关键不是“自动迭代”，而是“基于什么反馈迭代”。

## 2. 全自动循环最大的问题是没有真实反馈

视频里反复强调一个点：AI 不能只靠一份 Markdown 需求文档，就完整理解一个产品的灵魂。

这句话听起来像审美判断，但落到工程上，其实是反馈问题。

真实开发里，人类并不是一次性把所有判断都写进规格文档。很多判断是在过程中出现的：

- 这个交互是不是太绕。
- 这个边界情况是否应该特殊处理。
- 这个文案是否符合产品气质。
- 这个抽象是否会让后续维护更难。
- 这个实现是否只是“能跑”，但不是“能交付”。

这些判断很难提前全部写清楚。

如果 loop 里没有人类反馈，也没有可执行评测，模型只能基于自己的假设继续。它可能非常勤奋，但勤奋地走错路。

这也是为什么“让 Agent 自己跑一晚上”经常让人失望：第二天看到的不是完成品，而是一堆需要倒回去读的中间产物。

## 3. 有用的是受限反馈循环

我看完这条视频后，最想留下来的概念是：

```text
不要追求全自动 Agent Loop。
要设计受限反馈循环。
```

受限反馈循环至少有四个条件：

| 条件         | 作用                         |
| ------------ | ---------------------------- |
| 目标足够窄   | Agent 知道这一轮到底要改什么 |
| 反馈可验证   | 系统能判断改动有没有变好     |
| 停止条件明确 | 循环不会无限消耗预算         |
| 人类能接管   | 出现歧义时可以停下来问人     |

例如视频里提到的代码审查循环就比较合理：

```text
提交一个小 PR
  -> 代码审查 Agent 给出明确问题和评分
  -> 编码 Agent 读取审查意见并修复
  -> 再次提交
  -> 分数达到门槛后停止
```

这个 loop 之所以比“全自动做一个产品”可靠，是因为它有明确边界：

- 输入是一个 PR，不是整个产品愿景。
- 反馈来自代码审查，不是模型自我感觉。
- 门槛可以量化，例如达到某个评分。
- 单轮 diff 可以控制，人类还能读懂。

这才是 loop 的工程形态：不是把控制权全部交给模型，而是把一段可验证、可回滚、可停止的工作交给模型。

## 4. Loop Engineer 和前面几个 Engineer 的区别

前面写过一条笔记：[从 Prompt Engineer 到 Content Engineer 再到 Harness Engineer](/notes/ai/prompt-content-harness-engineer/)。

那篇里把 AI 应用工程拆成三层：

```text
Prompt Engineer  ->  Content Engineer  ->  Harness Engineer
优化问法             组织材料              搭建运行系统
```

看完这个视频后，可以把 Loop Engineer 放在 Harness Engineer 里面更靠后的一个位置。

| 角色             | 关心的问题             | 典型产物                                 |
| ---------------- | ---------------------- | ---------------------------------------- |
| Prompt Engineer  | 怎么问模型             | system prompt、few-shot、输出格式        |
| Content Engineer | 给模型什么材料         | 知识库、元数据、内容流程、RAG 语料       |
| Harness Engineer | 如何让模型可靠执行     | 工具、状态、权限、评测、追踪、人工确认   |
| Loop Engineer    | 如何让模型基于反馈迭代 | 反馈信号、停止条件、预算、回滚、循环策略 |

如果说 Prompt Engineer 解决的是一次调用的质量，Content Engineer 解决的是信息环境，Harness Engineer 解决的是运行外壳，那么 Loop Engineer 解决的是：

```text
当模型做完一步之后，系统凭什么决定下一步？
```

这个问题比“让模型继续”难得多。

因为继续之前要判断：

- 当前结果是否可接受。
- 失败是否可恢复。
- 继续是否值得花更多成本。
- 是否已经触碰权限边界。
- 是否应该交给人类判断。
- 这一轮产生的状态是否还能被信任。

Loop Engineer 不是写一个 `while` 循环。它是在设计循环里的判断权。

## 5. Loop 的输入必须小

视频里还提到一个很现实的规模问题：当一次提交或一次任务太大，Agent 的判断会迅速变差。

这个现象和上下文工程是一回事。

上下文太大时，模型会遇到几个问题：

- 关键约束被淹没。
- 相关文件和无关文件混在一起。
- 审查意见变得笼统。
- 修复动作影响面太大。
- 下一轮很难判断到底是哪次修改带来了变化。

所以一个可用的 loop，输入应该尽量小。

对代码任务来说，比较好的单位不是：

```text
帮我重构整个系统。
```

而是：

```text
把这个 PR 的类型错误修到通过 astro check。
```

或者：

```text
根据这 3 条 review comment 修改对应文件，保持行为不变。
```

小输入不是为了让 Agent 显得弱，而是为了让反馈有效。任务越小，评测越清楚，回滚越容易，人类越能判断质量。

## 6. Loop 的反馈要比模型更硬

全自动 loop 容易失控的另一个原因，是反馈也交给模型自己判断。

例如：

```text
你觉得自己完成了吗？
```

这个反馈太软。模型可能会给出一个很自信的“完成了”，但系统没有任何证据。

更硬的反馈包括：

- 测试是否通过。
- 类型检查是否通过。
- 构建是否通过。
- lint 是否通过。
- 审查 Agent 是否给出明确阻塞问题。
- 线上接口是否返回预期结果。
- 人类是否确认当前版本可接受。

在 AI 工程里，越靠近执行层，越不能只相信自然语言自评。

一个更可靠的 loop 应该长这样：

```text
执行动作
  -> 收集机器可验证结果
  -> 收集结构化审查意见
  -> 判断是否达到门槛
  -> 达不到才进入下一轮
```

这里的关键是：反馈必须能改变控制流。

如果测试失败，不能继续假装成功。如果 diff 太大，应该停下来请求人类 review。如果预算快用完，应该停止并总结当前状态。

## 7. 人类确认点不是倒退

很多 Agent 叙事会把 human-in-the-loop 说成不够先进，好像真正先进的系统应该完全不需要人。

看完这个视频后，我更倾向于反过来理解：

```text
人类确认点是高质量 loop 的一部分。
```

因为很多反馈不是机器可以稳定判断的。

机器擅长判断：

- 类型是否正确。
- 测试是否通过。
- 接口是否返回 200。
- 文件是否符合 schema。
- 代码是否违反某些静态规则。

人类更适合判断：

- 产品方向是否对。
- 取舍是否合理。
- 体验是否自然。
- 文案是否有味道。
- 这个修改是否值得继续投入。

所以 loop 不是要消灭人，而是要把人放在更有价值的位置。

低风险、可验证、重复性的部分可以自动循环。高风险、审美性、方向性的部分应该停下来确认。

## 8. 我会怎么用这套思路

如果把这条视频的观点落到日常 AI 编程，我会这样用：

第一，不让 Agent 接太大的目标。

不要一句话让它“重构整个项目”。先让它读现状、拆任务、确认边界。真正进入 loop 的任务应该足够小，最好能在一个 PR 或一个验证命令里闭环。

第二，把反馈写进流程。

例如：

```text
实现
  -> pnpm run check
  -> 根据错误修复
  -> 再次 check
  -> diff 自检
  -> 总结风险
```

这比“写完后你自己看看”可靠得多。

第三，设置停止条件。

常见停止条件可以很简单：

- 连续两次同类错误没有进展就停。
- diff 超过预期范围就停。
- 需要产品判断就停。
- 验证命令通过就停。
- 预算或时间超过上限就停。

没有停止条件的 loop，本质上是在把成本控制交给模型。

第四，小步提交。

如果一次改动超过人类能轻松 review 的范围，loop 就开始失去意义。因为 feedback loop 的质量取决于反馈者能不能看清变化。

## 9. 一个实践模板

可以把可用的 Agent Loop 写成这个模板：

```text
目标：
  修复一个明确问题，而不是完成一个模糊愿景。

输入：
  限定文件、限定 issue、限定 PR 或限定测试失败。

动作：
  Agent 可以读取、修改、运行验证命令。

反馈：
  测试、类型检查、构建、代码审查、人类确认。

门槛：
  必须达到哪个条件才算完成。

预算：
  最多循环几轮，最多改多少范围。

停止：
  通过、阻塞、超预算、需要人类判断。

记录：
  每轮做了什么，失败在哪里，下一步建议是什么。
```

这就是 Loop 工程的基本形状。

它不是一个很玄的 agent 理论，而是一组具体的工程问题：

- 怎么定义任务边界。
- 怎么收集反馈。
- 怎么控制预算。
- 怎么判断是否继续。
- 怎么让人类在正确的位置介入。

## 10. 总结

Agent Loop 最容易被误解成“让 AI 一直自动干活”。

但真正有价值的 loop，不是更长的自动驾驶，而是更短、更硬、更可验证的反馈循环。

把它和前面的几个角色放在一起看：

```text
Prompt Engineer 关心一次调用怎么问。
Content Engineer 关心模型基于什么材料回答。
Harness Engineer 关心模型如何在系统里可靠行动。
Loop Engineer 关心行动之后如何基于反馈继续、停止或交给人。
```

所以 Loop 工程的核心不是自动化程度，而是反馈质量。

一个坏 loop 会把错误自动化。

一个好 loop 会把反馈工程化。
