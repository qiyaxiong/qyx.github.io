前面几篇已经把模型应用和模型训练两条线串起来了：

- RAG 解决外部知识。
- 上下文工程解决信息组织。
- 工具调用解决外部动作。
- 工作流和 Agent 解决多步执行。
- MCP 解决工具生态接入。
- RLHF、DPO、GRPO 解决偏好和推理训练。

这一篇收束到 Agent 训练工程。

Agent 训练和普通问答训练不同。普通模型输出一段文本，Agent 输出的是一条执行轨迹。

```text
任务
  -> 计划
  -> 调用工具
  -> 观察结果
  -> 修改策略
  -> 继续执行
  -> 产生可验证结果
```

要训练或评测 agent，必须处理 rollout、沙箱、轨迹评测和分布式执行。

## 1. Agent 的训练对象是轨迹

一个问答样本长这样：

```json
{
  "question": "RAG 是什么？",
  "answer": "RAG 是检索增强生成..."
}
```

一个 agent 样本可能长这样：

```json
{
  "task": "修复构建失败并提交补丁",
  "trajectory": [
    { "type": "thought", "text": "先运行构建复现错误" },
    { "type": "tool", "name": "run_command", "input": "pnpm run build" },
    { "type": "observation", "text": "MDX frontmatter description too long" },
    { "type": "tool", "name": "edit_file", "input": "shorten description" },
    { "type": "tool", "name": "run_command", "input": "pnpm run build" },
    { "type": "observation", "text": "build passed" }
  ],
  "result": "success"
}
```

训练对象不只是最终答案，而是完整过程。

这带来几个难点：

- 轨迹长度不固定。
- 工具调用可能失败。
- 环境状态会变化。
- 中间步骤有好坏之分。
- 最终成功可能来自偶然。
- 失败也可能包含有价值尝试。

## 2. Rollout 是什么

Rollout 可以理解为让 agent 在环境里实际跑一遍任务。

```text
给定任务
  -> agent 自主执行
  -> 记录每一步
  -> 得到最终状态
  -> 评估成功或失败
```

在推理训练里，rollout 是生成候选解法；在 agent 训练里，rollout 是执行候选轨迹。

例如同一个任务可以采样多条轨迹：

```text
trajectory A: 直接改文件 -> 测试失败 -> 放弃
trajectory B: 先复现错误 -> 定位原因 -> 修复 -> 测试通过
trajectory C: 改错文件 -> 引入新问题
```

如果评测器能判断 B 成功，就可以把 B 作为正向训练信号。

## 3. 为什么需要沙箱

Agent 会调用工具，甚至修改文件、运行命令、访问网络。

训练和评测时不能让它直接操作真实生产环境。

沙箱的作用是：

- 隔离文件系统。
- 限制网络访问。
- 控制环境变量。
- 限制执行时间。
- 捕获工具输出。
- 回滚环境状态。
- 防止破坏真实数据。

一个代码 agent 的沙箱可能包含：

```text
临时 git worktree
隔离容器
只读密钥
受限网络
测试命令白名单
执行超时
产物收集
```

没有沙箱，就很难安全地让 agent 大规模 rollout。

## 4. 任务环境要可复现

评测 agent 时，环境必须可复现。

如果今天任务能过，明天依赖变了、网络变了、数据库状态变了，结果就不可比较。

可复现环境需要：

- 固定代码版本。
- 固定依赖版本。
- 固定测试数据。
- 固定初始文件状态。
- 固定权限和工具集合。
- 明确超时和资源限制。

例如：

```json
{
  "taskId": "fix_mdx_description_001",
  "repoCommit": "abc123",
  "setupScript": "pnpm install --frozen-lockfile",
  "verificationCommand": "pnpm run build",
  "timeoutMs": 600000
}
```

否则 agent 评测会变成不可重复实验。

## 5. 评测不只看最终输出

Agent 评测至少有三层：

```text
最终结果评测
轨迹评测
安全评测
```

最终结果评测：

- 测试是否通过。
- 文件是否符合要求。
- URL 是否返回 200。
- 任务是否完成。

轨迹评测：

- 是否先复现问题。
- 是否读了相关文件。
- 是否调用了不必要工具。
- 是否重复无效尝试。
- 是否正确处理错误。

安全评测：

- 是否访问无权限资源。
- 是否执行危险命令。
- 是否泄露密钥。
- 是否在写操作前确认。

只看最终结果不够。一个 agent 可能碰巧成功，但过程很危险。

## 6. 奖励设计怎么做

Agent 奖励可以分成稀疏奖励和稠密奖励。

稀疏奖励：

```text
任务完成 +1
任务失败 0
危险操作 -1
```

稠密奖励：

```text
复现错误 +0.1
读到相关文件 +0.1
修复测试 +0.5
无关工具调用 -0.05
重复调用 -0.1
```

稀疏奖励简单，但学习效率低。稠密奖励信息更多，但更容易被模型投机。

例如如果奖励“读文件数量”，agent 可能疯狂读文件。

所以奖励要围绕真实目标：

- 完成任务。
- 证据充分。
- 步骤经济。
- 安全合规。
- 可恢复。

## 7. 轨迹数据怎么存

Agent rollout 需要完整记录。

一条轨迹至少包含：

```json
{
  "runId": "run_123",
  "taskId": "task_456",
  "model": "agent-model-v1",
  "tools": ["read_file", "edit_file", "run_tests"],
  "steps": [
    {
      "index": 1,
      "type": "tool_call",
      "tool": "run_tests",
      "input": "pnpm run build",
      "outputSummary": "failed: MDX schema validation",
      "latencyMs": 12000
    }
  ],
  "finalStatus": "success",
  "verification": {
    "command": "pnpm run build",
    "exitCode": 0
  }
}
```

这些数据可以用于：

- 离线分析。
- 训练样本筛选。
- 错误归因。
- 回放调试。
- 评测对比。

没有轨迹数据，就谈不上 agent 训练工程。

## 8. 分布式执行为什么重要

Agent rollout 很贵。

一个任务可能要运行几十秒到几十分钟。如果要采样多个模型、多个温度、多个任务，就需要分布式执行。

基本架构：

```text
任务队列
  -> 调度器
  -> 多个 sandbox worker
  -> 轨迹存储
  -> 评测器
  -> 训练数据筛选
```

每个 worker 执行：

```text
拉取任务
  -> 初始化沙箱
  -> 运行 agent
  -> 收集轨迹
  -> 运行验证
  -> 上报结果
  -> 清理环境
```

这和普通批量生成文本不同，因为每个样本都可能产生外部动作。

## 9. 成本和限流

Agent 训练成本来自多个地方：

- 模型 token。
- 工具调用。
- 沙箱运行时间。
- 测试执行。
- 网络请求。
- 存储轨迹。
- 人工审核。

需要设置预算：

```json
{
  "maxSteps": 30,
  "maxToolCalls": 80,
  "maxRuntimeMs": 900000,
  "maxInputTokens": 200000,
  "maxOutputTokens": 50000
}
```

还要做限流：

- 单任务限流。
- 单用户限流。
- 单工具限流。
- 下游 API 限流。
- 全局并发限制。

否则一个失败循环可能烧掉大量资源。

## 10. 自动评测和人工评审结合

Agent 任务不是都能自动评测。

代码任务可以用测试；
数学任务可以验证答案；
网页任务可以截图和 DOM 检查；
写作任务则需要人工或模型辅助评审。

建议组合：

```text
自动硬指标
  -> 测试通过、命令退出码、URL 200、schema 校验

模型辅助评审
  -> 回答质量、引用支持、轨迹合理性

人工抽检
  -> 高风险任务、边界案例、评测器争议样本
```

不要完全依赖模型评测模型。它可以提高效率，但不能替代关键场景的人类基准。

## 11. 数据筛选比数据数量更重要

不是所有 rollout 都适合训练。

应该筛选：

- 成功且轨迹简洁的样本。
- 失败但能体现关键错误的样本。
- 从失败恢复成功的样本。
- 遵守权限和确认点的样本。
- 有明确验证证据的样本。

应该剔除：

- 碰巧成功但过程危险的样本。
- 重复调用工具的样本。
- 使用泄露信息的样本。
- 依赖不可复现环境的样本。
- 输出无法验证的样本。

Agent 训练不是把所有轨迹都丢进模型，而是构建高质量行为数据。

## 12. 和产品系统的闭环

上线后的 agent 还会产生真实轨迹。

这些轨迹可以进入改进闭环：

```text
线上任务
  -> 日志和轨迹
  -> 错误分类
  -> 构造评测用例
  -> 离线 rollout
  -> 筛选训练数据
  -> 更新模型或策略
  -> 回归评测
  -> 灰度上线
```

注意线上用户数据要做隐私保护和授权处理。

不能因为要训练，就无边界地收集和使用敏感轨迹。

## 13. 什么时候不需要训练 Agent

不是所有 agent 都需要自己训练模型。

很多场景用现成模型 + 好的工作流 + 工具设计 + 评测就够了。

需要训练的信号通常是：

- 任务量很大，人工修正成本高。
- 现成模型在固定任务上反复犯同类错误。
- 有稳定自动评测器。
- 能构造可复现沙箱。
- 有足够高质量轨迹数据。
- 业务收益能覆盖训练和运维成本。

如果没有这些条件，先优化提示、工具、RAG、工作流和评测，往往更划算。

## 14. 当前博客问答入口应该怎么演进

对于当前博客，合理路线不是马上训练 agent。

更稳的路线是：

```text
1. 完成 12 篇主线内容
2. 用 collection 统一系列入口
3. 把文章整理进 LLMWiki 或检索索引
4. 做博客问答 API
5. 记录问题、检索证据和引用
6. 做小评测集
7. 再考虑 agent 工具和训练
```

也就是说，先把知识层和检索链路打稳，再谈 agent 训练。

Agent 训练工程是后面的放大器，不是第一步。

## 15. 小结

Agent 训练工程关注的是完整执行系统：

- rollout：让 agent 在环境里实际执行任务。
- 沙箱：隔离风险，保证可复现。
- 轨迹：记录每一步工具调用和观察。
- 评测：同时看结果、过程和安全。
- 奖励：围绕真实目标设计，防止投机。
- 分布式执行：大规模采样和验证。
- 数据筛选：选高质量轨迹，而不是堆数量。

至此，这 12 篇主线从大模型生成 token 开始，经过 attention、tokenizer、RAG、上下文、工具、agent、MCP、RLHF、DPO/GRPO，最后落到 agent 训练工程。

如果把它们连起来看，会得到一条完整路径：

```text
理解模型如何生成
  -> 理解上下文为什么贵
  -> 学会把外部知识接进模型
  -> 学会把工具和流程接进模型
  -> 理解模型如何通过偏好和结果继续训练
  -> 走向可评测、可执行、可迭代的 agent 系统
```
