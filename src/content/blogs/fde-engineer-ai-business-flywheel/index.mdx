我看完这期视频后的结论很直接：

```text
AI 时代的 FDE，不是更像实施顾问，而是更像客户现场和模型飞轮之间的接口。
```

视频里谈到的 OpenAI、Ramp、Nominal、Dataland，虽然业务不同，但都在讲同一件事：模型能力变强之后，真正稀缺的不是“再写一层胶水代码”，而是把业务问题、评测标准、客户反馈和后训练连成闭环的人。

## 1. 旧 FDE 和新 FDE 不一样

传统 FDE 更像是把产品卖进客户环境里的人：

- 接需求。
- 配环境。
- 跑通集成。
- 修权限和网络。
- 把 demo 变成可交付方案。

这类工作重要，但边界比较清楚。客户的问题通常是“怎么接上”。

AI 时代的 FDE，问题变成了“接上之后怎么让它真的好用”。

这时 FDE 要处理的不是单纯集成，而是：

- 哪些场景值得做成产品。
- 哪些失败应该被定义成 eval。
- 哪些客户反馈是噪声，哪些是模型短板。
- 哪些流程应该保留人工，哪些可以自动化。
- 哪些 case 应该反馈给研究团队，变成后训练数据。

这已经不是纯交付了，而是把客户现场变成产品输入。

## 2. AI 越强，FDE 越重要

很多人直觉上会觉得：模型越强，越不需要 FDE。

实际相反。

模型越强，通用能力越高，但行业问题越暴露出三个难点：

1. 业务语义是碎的，不在公开语料里。
2. 客户流程是脏的，不符合标准化假设。
3. 成败标准不是“回答像不像”，而是“结果有没有用”。

所以 FDE 的价值不是补模型短板的低级工程，而是把行业知识翻译成模型可以学习和执行的东西。

这包括：

- 把客户目标拆成可测的任务。
- 把成功和失败写成 eval。
- 把真实样本整理成训练材料。
- 把流程中的人工步骤找出最小可自动化单元。

模型越强，越需要有人把“强模型”落到“真实业务”上。

## 3. FDE 的核心产物不是代码

这期视频里最值得抠的一点，是 FDE 的价值不止在实现。

如果只看代码，FDE 很容易被误解成高级驻场。
真正的产物通常是这些东西：

| 产物         | 作用                       |
| ------------ | -------------------------- |
| `eval`       | 定义什么叫好、什么叫坏     |
| 业务流程抽象 | 把客户需求变成稳定任务     |
| 数据样本     | 让模型看到真实分布         |
| 反馈循环     | 把客户现场变成产品输入     |
| 可复用能力   | 把一次交付沉淀成下一次起点 |

代码只是外壳。真正值钱的是把问题定义清楚。

这也解释了为什么优秀的 FDE 往往不是纯后端、纯前端，甚至也不只是 AI 工程师，而是懂业务、懂产品、懂工程、还能快速做取舍的人。

## 4. FDE 和咨询的分界线在经济模型

视频里一个判断很重要：如果客户依赖的是“人”，不是“产品”，那就会滑向咨询。

区别不在于你有没有在客户现场，而在于你有没有沉淀出可复用资产。

可以这样看：

| 模式                                 | 结果       |
| ------------------------------------ | ---------- |
| 一次性交付，反复人工跟进             | 像咨询     |
| 为一个客户做的能力能复用到多个客户   | 像软件     |
| 每次都得重新解释、重新搭建、重新救火 | 规模会卡住 |

AI 时代的 FDE 如果只是“帮客户把模型跑起来”，会很快变成高价外包。

如果 FDE 能把客户问题沉淀成产品能力、评测框架或后训练数据，它就在帮公司构建飞轮。

## 5. OpenAI 式飞轮的关键

视频里提到的一个方向我很认同：

```text
客户现场 -> 失败样本 -> eval -> 合成数据 -> post-training -> 更强模型 -> 再回到客户现场
```

这不是线性项目，而是回路。

FDE 的价值就在回路中间：

- 他们最先看到真实失败。
- 他们知道失败是业务问题还是模型问题。
- 他们能把问题翻译成训练信号。
- 他们能判断哪些改动值得继续投资源。

这件事比单纯集成更像“业务版的 harness 工程”。

Harness Engineer 关心模型怎么在系统里稳定执行，FDE 关心模型怎么在真实客户业务里变得更强。

## 6. 一个好的 FDE 要像什么人

这类角色的要求很苛刻，因为它本质上是复合型角色。

我会把它概括成四个词：

- `technical depth`
- `customer judgment`
- `product sense`
- `ownership`

少一个都不够。

只会写代码，不够，因为你不知道什么值得做。
只会聊客户，不够，因为你拿不出可执行方案。
只会做 demo，不够，因为你没有把失败闭环起来。
只会做定制，不够，因为你没有产品化思维。

最好的 FDE 往往是那种能同时判断“这单值不值得接”和“这个方案能不能沉淀”的人。

## 7. 对个人意味着什么

如果你关心 AI 时代的职业路径，FDE 这条线其实在说明一件事：

```text
AI 工程能力正在从“写出模型旁边的代码”转向“定义系统边界和业务闭环”。
```

所以更有价值的能力会是：

- 会把模糊需求拆成任务。
- 会把业务结果转成评测标准。
- 会识别模型能力和业务约束之间的差距。
- 会把一次交付沉淀成可复用的资产。
- 会知道什么时候该继续做产品，什么时候该停下来问客户真正要什么。

这和前面几篇文章里讲的上下文工程、工具调用、工作流、Harness 工程，其实是同一条主线。

FDE 只是把这条主线推到了客户现场。

## 8. 总结

AI 时代的 FDE 不是“会 AI 的实施工程师”这么简单。

它更像一个接口角色：

- 向内连接产品、模型和训练。
- 向外连接客户、业务和收入。

如果做得好，FDE 不是成本中心，而是飞轮的一部分。

如果做得不好，它就会退化成高价咨询。

这条线真正的分水岭，不是你有没有在现场，而是你能不能把现场变成可复用的产品能力。
