很多人第一次接触大模型 API 时，会自然地把“上下文长度”理解成“字数”。

这很容易造成误判。因为模型真正处理的不是字，也不是词，而是 token。一个中文句子、一段代码、一张 Markdown 表格、一份 JSON 配置，看起来长度可能差不多，但拆成 token 后的数量会很不一样。

理解 tokenizer，才能理解为什么长上下文贵，为什么代码和表格有时特别费 token，为什么 RAG 要切 chunk，为什么 agent 的工具返回值不能无限塞进 prompt。

这篇文章接着上一篇 Attention 与 KV cache，专门解释 token、tokenizer 和上下文成本之间的关系。

## 1. Token 是什么

大模型不能直接处理原始字符串。输入文本会先经过 tokenizer，被拆成一串 token id。

简化地看：

```text
原始文本
  -> tokenizer
  -> token ids
  -> embedding
  -> transformer
  -> 输出下一个 token
```

例如一句话：

```text
今天天气很好，我们去散步。
```

并不一定会被拆成：

```text
今 / 天 / 天 / 气 / 很 / 好 / ...
```

也可能是：

```text
今天 / 天气 / 很好 / ， / 我们 / 去 / 散步 / 。
```

具体怎么拆，取决于模型使用的 tokenizer。不同模型、不同厂商、不同版本，tokenizer 可能都不完全一样。

所以严格地说，不能用“字符数”精确推导 token 数，只能估算。

## 2. Tokenizer 为什么不是简单分词

传统中文分词关心的是语义词：

```text
我 / 想 / 查询 / 订单 / 状态
```

Tokenizer 关心的是模型词表里的可复用片段。它通常会把常见片段压缩成较少 token，把罕见片段拆成更多 token。

这类方法背后的直觉是：

- 常见文本片段应该更容易表示。
- 罕见字符、符号组合、陌生格式可以继续拆小。
- 词表大小有限，不可能每个词、每个变量名、每个 URL 都单独占一个 token。

因此，tokenizer 不是自然语言分词器。它更像是模型输入层的压缩编码器。

## 3. 上下文成本到底在算什么

当我们说“上下文成本”，通常包含几类成本：

| 成本 | 含义 |
| --- | --- |
| 输入 token 成本 | prompt、系统提示词、历史消息、检索文档、工具结果 |
| 输出 token 成本 | 模型生成的回答 |
| 延迟成本 | token 越多，处理越慢 |
| 注意力成本 | 长上下文会增加 attention 计算量 |
| KV cache 成本 | 已处理上下文要缓存 key/value，长对话更占资源 |
| 工程复杂度 | 需要截断、摘要、检索、缓存和权限控制 |

API 价格通常按输入 token 和输出 token 计费，但工程里的真实成本不止价格。

一个 agent 如果每轮都把全部历史、全部工具结果、全部文档塞回模型，就算单轮能跑通，长时间使用也会越来越慢、越来越贵、越来越不稳定。

## 4. 中文为什么和英文不一样

英文里空格天然分隔单词：

```text
The model uses tokens to represent text.
```

很多英文常见词或词根会被 tokenizer 合并成较少 token。

中文没有空格：

```text
模型使用 token 表示文本。
```

中文字符和常见词也可以被合并，但合并方式更依赖词表覆盖情况。常见中文短语可能比较省 token，罕见术语、混合中英文、特殊符号就可能被拆得更多。

例如下面这些文本，字符数接近，但 token 数不一定接近：

```text
模型上下文窗口
LLM context window
模型 context window
```

混合语言时尤其明显。因为 tokenizer 可能对英文词、中文词、空格、标点分别处理。

一个实用经验是：不要假设“中文一定比英文贵”或“英文一定比中文省”。要看具体 tokenizer 和具体文本形态。

## 5. 代码为什么常常费 token

代码看起来结构清晰，但对 tokenizer 来说可能很碎。

例如：

```ts
async function fetchUserProfile(userId: string) {
  const response = await fetch(`/api/users/${userId}/profile`)
  return response.json()
}
```

这里有很多自然语言里不常见的片段：

- 驼峰变量名：`fetchUserProfile`
- 类型标注：`: string`
- 模板字符串：`` `/api/users/${userId}/profile` ``
- 符号：`{`、`}`、`()`, `=>`, `;`
- 缩进和换行

Tokenizer 可能会把一个长变量名拆成多个 token：

```text
fetch / User / Profile
```

也可能把路径拆成很多片段：

```text
/ / api / / users / / ${ / user / Id / } / / profile
```

所以代码上下文的成本往往不只取决于“行数”，还取决于命名风格、符号密度、路径、泛型、嵌套结构和注释量。

这也是为什么让模型读完整仓库通常不现实。更好的方式是先检索相关文件，再读取局部代码，再让模型基于证据分析。

## 6. JSON 为什么也很贵

JSON 是 agent 和工具调用里最常见的格式，但它并不一定省 token。

例如一个工具返回：

```json
{
  "user_id": "usr_9f83a7",
  "subscription_status": "active",
  "latest_invoice_id": "inv_202606110001",
  "feature_flags": {
    "rag_enabled": true,
    "agent_beta_access": true,
    "billing_v2": false
  }
}
```

这里很多 token 花在结构上：

- 引号
- 冒号
- 逗号
- 下划线字段名
- 布尔值
- 嵌套对象
- 长 id

JSON 的优点是结构清楚，缺点是冗余明显。

如果工具返回值只是给模型阅读，有时可以把 JSON 压缩成更短的自然语言摘要：

```text
用户 usr_9f83a7 当前订阅为 active，已开启 RAG 和 agent beta，未开启 billing v2。
```

但如果后续还要程序解析，JSON 更安全。

工程上要区分两类输出：

| 用途 | 推荐形式 |
| --- | --- |
| 给模型阅读 | 精简摘要、表格、要点 |
| 给程序解析 | JSON、schema、结构化返回 |

不要把完整数据库记录、完整 API 响应、完整日志直接塞进 prompt。先筛字段，再摘要。

## 7. 表格为什么容易膨胀

Markdown 表格很适合人读，但对 tokenizer 不一定友好。

例如：

```md
| 模块 | 输入 | 输出 | 风险 |
| --- | --- | --- | --- |
| 检索 | 用户问题 | chunk 列表 | 召回不准 |
| 重排 | chunk 列表 | topK 证据 | 延迟增加 |
| 生成 | 证据 + 问题 | 答案 | 幻觉 |
```

表格里有大量重复符号：

```text
| | | --- | --- |
```

如果表格很宽、行很多、单元格里还有长文本，token 会快速增长。

给模型看的表格可以改成列表：

```text
检索：输入用户问题，输出 chunk 列表，主要风险是召回不准。
重排：输入 chunk 列表，输出 topK 证据，主要风险是延迟增加。
生成：输入证据和问题，输出答案，主要风险是幻觉。
```

这个版本对模型同样清楚，而且通常更省上下文。

## 8. 长文档为什么不能直接塞进 prompt

假设一篇文档有 20,000 个 token，而用户只问：

```text
虚拟商品能退款吗？
```

如果直接把整篇文档塞给模型，问题很明显：

- 大量无关内容占用上下文。
- 模型需要在长文档里自己找答案。
- 延迟和成本都变高。
- 如果文档里有多个相似段落，模型可能引用错位置。

RAG 的基本思想就是反过来：

```text
先检索相关片段
再把少量证据交给模型
```

也就是：

```text
问题
  -> 检索 topK chunks
  -> 组装上下文
  -> 模型回答
```

上下文窗口越大，越不代表应该把所有东西都塞进去。大窗口更像安全余量，不是浪费空间的理由。

## 9. Chunk 大小怎么影响 token 成本

RAG 里最常见的问题之一是 chunk 切多大。

如果 chunk 太大：

- 每次召回带入的 token 多。
- 相关信息被无关内容稀释。
- 多个 chunk 加起来容易超上下文。

如果 chunk 太小：

- 语义不完整。
- 模型看不到上下文。
- 需要召回更多 chunk 才能回答。

一个可用的起点：

```text
普通中文知识库：500 到 1000 个中文字符一个 chunk
技术文档：按小节切，保留标题层级
代码文档：按函数、类、文件片段切
表格文档：优先按行组或业务对象切
```

更重要的是每个 chunk 要带 metadata：

```json
{
  "title": "退款规则说明",
  "section": "虚拟商品退款",
  "content": "虚拟商品一经交付...",
  "url": "/docs/refund-policy",
  "chunkIndex": 3
}
```

这样模型不只拿到正文，还知道这段内容来自哪里。

## 10. Agent 为什么更需要控制上下文

普通聊天通常是：

```text
用户问题 -> 模型回答
```

Agent 往往是：

```text
用户目标
  -> 规划
  -> 调工具
  -> 读文件
  -> 搜索
  -> 再调工具
  -> 汇总
  -> 回答
```

每一步都会产生上下文：

- system prompt
- developer prompt
- 用户消息
- 历史对话
- 工具 schema
- 工具调用参数
- 工具返回值
- 检索证据
- 中间推理摘要
- 最终回答

如果不控制，agent 的上下文会膨胀得很快。

所以 agent 工程里要做几件事：

1. 工具返回值要短，只返回必要字段。
2. 大结果要分页，不要一次返回全部。
3. 搜索结果先返回摘要，必要时再 read 详情。
4. 历史对话要摘要或裁剪。
5. 文件读取要按范围读，不要整仓库塞入。
6. RAG 证据要去重、排序、限量。

一个好的 agent 不是“什么都记住”，而是“知道什么时候该读什么”。

## 11. LLMWiki 对 token 成本的价值

如果网站后续要接 agent，只靠 HTML 页面或 Pagefind 不够。

更合理的是把内容导出成 LLMWiki 形式：

```text
llms.txt
llms-full.txt
wiki/index.json
wiki/pages/<slug>.md
wiki/graph.json
```

这对 token 成本有几个好处：

- agent 先读 index，而不是整站内容。
- agent 可以按 slug 精确读取文章。
- Markdown 比 HTML 更干净，少掉样式和布局噪声。
- wiki graph 可以帮助 agent 找相关文章，而不是盲目搜索。
- 每篇文章可以预先切出摘要、章节和引用信息。

对于博客问答入口，理想流程是：

```text
用户在文章页提问
  -> agent 收到当前 pageId
  -> read_wiki_page(pageId)
  -> 必要时 search_wiki(query)
  -> 读取相关页
  -> 回答并给出处
```

这样 agent 不需要把整个网站塞进 prompt，而是按需读取知识。

## 12. 如何估算 token 数

最准确的方式当然是用目标模型对应的 tokenizer。

工程上可以分三层：

### 第一层：粗估

在没有 tokenizer 的情况下，可以用经验值粗略估算：

```text
英文：一个 token 大致对应 3 到 5 个字符
中文：一个 token 可能接近 1 到 2 个汉字，也可能被常见词合并
代码/JSON：符号和变量名会让 token 数上升
```

这只能用于早期设计，不能用于精确计费或截断。

### 第二层：接入 tokenizer

在服务端用目标模型的 tokenizer 计算：

```ts
const tokenCount = countTokens(text, model)

if (tokenCount > maxInputTokens) {
  throw new Error('context too large')
}
```

这个能力应该放在服务端，不要只依赖前端估算。

### 第三层：运行时预算

每次调用模型前，都应该有一个预算：

```text
系统提示词：800 tokens
历史摘要：1000 tokens
当前用户问题：300 tokens
检索证据：6000 tokens
工具结果：1500 tokens
预留输出：2000 tokens
```

然后按优先级裁剪：

```text
保留系统提示词
保留当前问题
保留高相关证据
压缩历史
删除低相关证据
限制工具结果
预留输出空间
```

不要等 API 报错之后再处理上下文超限。

## 13. 减少 token 浪费的实用方法

### 删除样式噪声

不要把 HTML、CSS class、导航栏、页脚、广告、评论区一起喂给模型。

把页面转成干净 Markdown：

```text
标题
摘要
正文
引用链接
```

### 压缩工具结果

工具返回给模型的内容应该是“决策需要的信息”，不是“数据库完整对象”。

不好：

```json
{
  "user": { "...": "完整用户对象" },
  "orders": [{ "...": "完整订单对象" }],
  "logs": [{ "...": "500 行日志" }]
}
```

更好：

```json
{
  "summary": "用户有 1 个未支付订单和 2 个已完成订单。",
  "relevantOrders": [
    {
      "id": "ord_123",
      "status": "pending_payment",
      "amount": 199
    }
  ]
}
```

### 给长内容做分层

长文档不要只有一种形态。可以同时准备：

```text
全文
章节摘要
chunk
关键词
引用 URL
```

Agent 先读摘要，不够再读 chunk，仍然不够再读全文局部。

### 控制历史对话

长对话里，早期消息并不总是同等重要。

可以维护：

```text
最近 N 轮原文
长期用户偏好摘要
任务状态摘要
已确认事实列表
```

这样比“把所有聊天记录一股脑塞回去”更稳定。

## 14. 一个上下文预算示例

假设模型输入上限是一个固定窗口，我们可以这样规划：

| 模块 | 预算 | 说明 |
| --- | --- | --- |
| System prompt | 固定小预算 | 角色、边界、安全规则 |
| Developer prompt | 固定小预算 | 产品规则、输出格式 |
| 用户问题 | 动态 | 当前轮必须保留 |
| 历史摘要 | 中等预算 | 保留任务状态，不保留全部原文 |
| 检索证据 | 最大预算 | RAG 的主要信息来源 |
| 工具结果 | 严格限制 | 只保留必要字段 |
| 输出预留 | 固定预留 | 防止回答被截断 |

伪代码：

```ts
const budget = {
  maxInput: 24000,
  reserveOutput: 3000,
  system: 1000,
  history: 2000,
  tools: 2000,
  evidence: 16000
}

const availableForInput = budget.maxInput - budget.reserveOutput

const evidence = selectEvidence({
  chunks,
  maxTokens: budget.evidence,
  dedupeBy: 'documentId'
})

const messages = buildMessages({
  systemPrompt,
  historySummary,
  userQuestion,
  evidence
})
```

重点不是这些数字本身，而是“每个部分都有预算”。没有预算的上下文管理，迟早会变成随机截断。

## 15. 对 RAG 和 Agent 的结论

Tokenizer 不是一个底层细节，而是 RAG 和 agent 设计的核心约束。

几个结论可以直接落地：

- 不要用字符数判断上下文成本。
- 不要把整篇长文档直接塞进 prompt。
- 代码、JSON、表格、日志都要特别注意 token 膨胀。
- 工具返回值要面向模型阅读重新设计。
- RAG chunk 要带标题、来源、章节和 URL。
- Agent 要有 search/read 两阶段，而不是一次性读取所有资料。
- LLMWiki 比 HTML 更适合作为 agent 的知识输入层。
- 每次模型调用前都应该做 token 预算。

上下文窗口越来越大，但工程上真正重要的不是“能塞多少”，而是“该塞什么”。

能控制 token，才能控制成本、延迟和回答质量。
